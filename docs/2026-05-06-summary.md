## 3. 잔여 HIGH_RISK 의 root cause 진단

django HIGH_RISK 의 STRONG evidence 2건 분석:

### Evidence 1: DEF-005 in vendored xregexp.js
파일: `django/.../static/admin/js/vendor/xregexp/xregexp.js:L3197`
스니펫: `exec(object);`

→ **vendor 제외로 해결됨** (위 1차 추가).

### Evidence 2: SP-003 in cache backend
파일: `django/core/cache/backends/filebased.py:L38`
시퀀스: `zlib.decompress` → `zlib.compress` → `pickle.loads`

→ Django 의 표준 파일시스템 캐시 백엔드. **완전히 합법**.
→ SP-003 (Encoded payload execution) sequence_patterns 가 zlib + pickle 콤보에 false-positive.
→ **Fix 위치는 sequence_patterns.py — Fix-1·2·4 범위 밖**.

### SP-003 정확한 Fix 안 (3차 범위 밖, 향후 작업)
1. SP-003 시퀀스 정의에 INFO_READ source 차원을 _필수_ 로 추가 (현재 ENCODE → EXEC 만으로 fire)
2. 즉 진짜 악성 패턴: `os.environ.get` (info) → `base64.b64encode` (encode) → `requests.post` (transmit) — 4 dimension
3. 합법 cache 패턴: `zlib.decompress` (decode) → `pickle.loads` (exec) — 2 dimension only, info source 없음
